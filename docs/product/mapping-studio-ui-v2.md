# CanonBridge Mapping Studio UI V2 - Sources, Outbound APIs, Credentials, and External Systems

**Version**: 2.0  
**Last Updated**: May 11, 2026  
**Status**: Product and UI Design Specification  
**Related architecture**: [Architecture V7 - Outbound API Calling and Credential Store](../architecture/architecture-v7-outbound.md)

> This document updates Mapping Studio for a world where an integration can start from Kafka, a webhook, scheduled external API polling, or manual data upload. It also adds credential management and outbound API health views while preserving the core promise: users do not need to know JSONata.

---

## 1. Product Principle

```text
The user describes intent.
Mapping Studio asks precise questions.
The platform generates configuration, JSONata, validation gates, and runtime calls in the background.
Secrets never appear in generated mapping code.
```

V2 keeps Mapping Studio no-code:

- Users choose source type visually.
- Users configure external APIs through forms.
- Users add credentials through a secure drawer.
- Users test real calls from the UI.
- Users map fields from JSON trees, not from JSONata.
- JSONata remains a generated runtime artifact.

## 3. Wizard Roadmap

The wizard remains a guided five-step flow.

```text
1. Source and trigger
2. Canonical schema
3. Field mapping
4. Resolve gaps
5. Test, validate, publish
```

V2 changes Step 1, Step 3, and Step 5.

| Step | V2 change |
|---|---|
| Step 1 | Source type cards and outbound/API credential setup. |
| Step 3 | Source tree can come from Kafka sample, webhook sample, external API response, or manual JSON. |
| Step 5 | Test can call a real external API, show HTTP errors, and then run mapping validation. |

## 4. Step 1 - Source and Trigger

### 4.1 Purpose

Step 1 answers:

```text
How does this transformation flow start?
Where does the example source JSON come from?
Do we receive data, poll data, or manually load data?
```

### 4.2 Layout

```text
+----------------------------------------------------------------------------+
| Source and trigger                                                          |
| Choose how CanonBridge receives or fetches data.                            |
|                                                                            |
| + Kafka Topic --------+ + Webhook -----------+ + External API -----------+  |
| | Stream from Kafka   | | They send to us    | | We go and fetch data    |  |
| | Existing pipeline   | | Unique URL + key   | | URL, method, auth       |  |
| +---------------------+ +--------------------+ +-------------------------+  |
|                                                                            |
| + Manual Data Upload -----------------------------------------------------+ |
| | Paste or drop JSON for test-only setup                                  | |
| +------------------------------------------------------------------------+ |
|                                                                            |
| Dynamic panel for selected source type                                      |
+----------------------------------------------------------------------------+
```

### 4.3 Source type cards

Each source card is a button-like selectable card.

Common card requirements:

- Icon
- Title
- One-line description
- Best-for hint
- Selected state
- Disabled state with reason
- Keyboard accessible
- `aria-pressed` for selected state

Source type enum:

```ts
type SourceType =
  | 'KAFKA_TOPIC'
  | 'WEBHOOK'
  | 'EXTERNAL_API_POLL'
  | 'MANUAL_UPLOAD';
```

### 4.4 Kafka Topic panel

Shown when `Kafka Topic` is selected.

Fields:

| Field | Control | Required | Notes |
|---|---|---:|---|
| Topic | searchable select | yes | List allowed raw topics for tenant. |
| Consumer group | text input | yes | Default generated from partner/event. |
| Sample source | segmented control | yes | `Upload JSON`, `Use last message`, `Import from DLQ`. |
| Schema version | select | yes | Defaults to latest input schema. |

Actions:

- `Fetch latest sample`
- `Upload sample JSON`
- `Analyze source`

### 4.5 Webhook panel

Shown when `Webhook (external service sends to us)` is selected.

Fields and generated values:

| Item | Control | Behavior |
|---|---|---|
| Webhook URL | read-only text + copy button | Generated after draft is saved. |
| Inbound API key | masked text + copy once | Generated by Credential Store / webhook secret manager. |
| Rotate key | button | Requires `webhook:secret_rotate`. |
| Expected content type | select | `application/json` for MVP. |
| Sample capture window | button | Opens a temporary listener state for showing the next received payload. |

Webhook URL example:

```text
https://api.canonbridge.example/webhooks/acme/order-created/wh_7R9k2
```

Security copy:

```text
Use this URL and API key in the external system. The key is shown once. You can rotate it later.
```

Error states:

- Webhook not generated yet: save draft first.
- API key already copied: show masked value and rotate action.
- Last delivery failed: show HTTP status and error body preview.

### 4.6 External API Poll panel

Shown when `External API - we fetch data` is selected.

Fields:

| Field | Control | Required | Notes |
|---|---|---:|---|
| Connection name | text input | yes | Example: `Carrier A orders`. |
| Method | segmented control | yes | `GET`, `POST`, `PUT`; `DELETE` not supported for source polling. |
| URL | URL input | yes | Must pass allowlist validation. |
| Query params | key/value table | no | Supports visual template tokens. |
| Headers | key/value table | no | Auth headers are locked when credential is selected. |
| Body | JSON editor or form builder | conditional | Required for POST/PUT. |
| Schedule | interval/cron selector | yes for polling | Example: every 5 minutes. |
| Checkpoint mode | select | yes | `watermark`, `page_token`, `idempotency_only`. |
| Timeout | numeric input | yes | Default 3000ms. |
| Retry policy | compact form | yes | attempts, backoff, retry statuses. |
| Authentication | credential selector | optional | Add or reuse credential. |

Primary actions:

- `Add authentication`
- `Test API call`
- `Use response as sample`
- `Analyze response tree`

### 4.7 Manual Data Upload panel

Manual data upload is for tests and setup only.

Controls:

- JSON textarea
- Strong drag/drop upload cards
- `.json` and `.txt` accepted
- Input schema upload
- Analyze payload

This is the current local/demo mode and should stay available.

- The draft stores only `credentialRef`.

## 6. External API Request Builder

### 6.1 Request editor sections

The External API panel is a vertical form with these sections:

```text

Authentication
Headers
Query parameters
Body
Timeout and retry
Response handling
Schedule and checkpoint
```

Each section is collapsible after completion.

### 6.2 URL validation

Validation rules:

- Must be HTTPS in production.
- Host must be in tenant allowlist.
- Private IP ranges blocked.
- Template tokens allowed only in path/query, not host.
- Redirects disabled by default.

Invalid URL message:

```text
This host is not allowed for outbound calls in this environment.
```

For query, headers, and body fields, users can insert tokens from source fields.


Token picker shows:

- Source JSON tree
- Trigger metadata

- Already mapped canonical fields, when available

Users never type JSONata tokens. The UI inserts safe template tokens.

## 7. Step 3 - Field Mapping with External API Sources

### 7.1 Source field panel behavior

The source field panel changes based on source type.

| Source type | Tree root |
|---|---|
| Kafka Topic | Uploaded/fetched Kafka sample payload |
| Webhook | Captured webhook sample payload |
| External API Poll | Latest successful API test response |
| Manual Upload | Uploaded/pasted JSON |

For external API sources, the tree label should be:

```text
External API response
```

If no API response sample exists:

```text
Run "Test API call" in Step 1 to generate a response tree.
```

### 7.2 External response tree

Tree node content:

```text
orders[]                 array · 12 items
  id                     string · "ORD-123"
  customer.name          string · "Ayse Yilmaz"
  total                  number · 98.4
```

Interactions:

- Click field to select source.
- Drag field to target.
- Filter by type, required/optional, array, unmapped.
- Show sample values.
- Show response status and API call timestamp above tree.

### 7.3 Outbound enrichment in Step 3

Some flows start from Kafka/webhook but need enrichment from an external API during mapping. In this case Step 3 shows an additional panel:

```text
Enrichment calls
+ shippingQuote   Success sample available   [Edit] [Retest]
+ customerLookup  Missing sample             [Configure]
```

Rules:

- Enrichment call output appears under tree root `outbound.<callId>.body`.
- Users map from the outbound tree like any other source.
- The UI labels the source clearly:

```text
Source: External call "shippingQuote"
```

### 7.4 No-code transform rule requirement

All existing visual transforms still apply:

- Direct map
- Format date
- Enum map
- Default value
- Combine fields
- String operations
- Array operations
- Math operations
- Conditional value
- Template string

When a user maps from outbound response data, the generated JSONata reads response context only:

```jsonata
outbound.shippingQuote.body.etaDays
```

It does not call the API.

## 8. Step 5 - Test, Validate, Publish

### 8.1 Test button behavior

Button label:

```text
Run test
```

Behavior by source type:

| Source type | Test behavior |
|---|---|
| Kafka Topic | Use selected sample or fetched non-prod message. |
| Webhook | Use captured sample or uploaded sample. |
| External API Poll | Make real API call, use response as source, run mapping. |
| Manual Upload | Use pasted/uploaded JSON. |

### 8.2 Test sequence for External API

```text
1. Validate URL, auth reference, timeout/retry config.
2. Resolve credential through Credential Store.
3. Execute external API call through outbound-call-manager.
4. If HTTP call fails, stop and show API error panel.
5. If HTTP call succeeds, parse response.
6. Validate input schema when configured.
7. Run generated JSONata against response context.
8. Validate canonical schema.
9. Show diff/output and publish readiness.
```

### 8.3 API error panel

When API test fails, show:

```text
External API call failed
Status: 503 Service Unavailable
Duration: 1200 ms
Attempts: 3
Retry policy: exhausted

Safe response body:
{ "error": "maintenance" }

[Edit request] [Retest] [Open External Systems]
```

Rules:

- Authorization headers are never shown.
- Body is masked.
- Show correlation ID.
- Show whether error is retryable.
- Link to Outbound Call History filtered to this call.

### 8.4 Success panel

When API call and mapping pass:

```text
Test passed
External API: 200 OK · 183 ms
Mapping: 4 ms
Canonical schema: passed
Required fields: 17 / 17
```

Actions:

- Save response as fixture.
- Save transformed output as expected fixture.
- Publish when approval rules pass.

## 12. Validation Rules

### 12.1 Source step validation

| Source type | Required before Next |
|---|---|
| Kafka Topic | Topic selected and sample available. |
| Webhook | Webhook generated and at least one sample captured or uploaded. |
| External API Poll | URL, method, schedule, timeout/retry valid, test response available. |
| Manual Upload | Valid JSON sample available. |

### 12.2 Credential validation

- Credential name is required.
- Auth type is required.
- Secret fields are required on create.
- Secret fields are write-only after save.
- Header name cannot be `Authorization` for API key unless explicitly allowed by admin.
- OAuth2 Token URL must pass outbound allowlist.
- Credential environment must match draft environment.

### 12.3 Publish gates for outbound mappings

Before publish:

- All referenced credentials are active.
- Last test for required external API source succeeded.
- Response sample is saved.
- Mapping validation passed.
- Canonical schema validation passed.
- No unresolved required target fields.
- User has publish permission.
- Audit event emitted.

## 16. Visual Design Requirements

### 16.1 Source cards

Cards must:

- Use 8px radius.
- Have clear selected state.
- Avoid nested cards inside cards.
- Use icon + title + concise detail.
- Be scannable in light and dark mode.
- Not rely only on color.

### 16.2 Credential drawer

The drawer must:

- Open from the right.
- Keep primary action sticky at bottom.
- Clearly mark secret fields as write-only.
- Show validation inline.
- Never show saved secret values.

### 16.3 External Systems dashboard

The dashboard should be operational and dense, not a marketing page:

- Table-first layout.
- Compact metric tiles.
- Filter bar.
- Detail drawer.
- Latency and success charts.
- Clear status labels.

### 16.4 Upload and drag/drop

All file uploads in Mapping Studio should use the stronger V2 upload pattern:

- Clickable dropzone/card.
- Drag-over state.
- Loaded state.
- Accepted file type label.
- Clear error when type or JSON parsing fails.

## 17. Generated Runtime Artifact

Published mapping package includes:

```text
config.json
input.v1.schema.json
canonical.v1.schema.json
inbound.v3.jsonata
outbound/request-templates/*
outbound/wsdl/*
fixtures/*
```

`config.json` contains:

- source type
- trigger configuration
- outbound call definitions
- credential references
- retry/timeout policies
- masking rules

`config.json` does not contain:

- API key values
- Basic auth passwords
- Bearer tokens
- OAuth2 client secrets
- OAuth2 access tokens

## 18. Acceptance Criteria

Mapping Studio UI V2 is complete when:

```text
1. User can choose Kafka, Webhook, External API, or Manual Upload in Step 1.
2. User can configure an external API without writing code.
3. User can create API Key, Basic Auth, Bearer Token, and OAuth2 credentials from a secure drawer.
4. Saved credentials are reusable by name and secrets are never displayed again.
5. User can test an external API and use the response as the source tree.
6. Step 3 field mapping works from external API response fields and enrichment response fields.
7. Step 5 runs the real API call before transformation and shows HTTP errors clearly.
8. User can distinguish upstream errors from mapping/schema errors.
9. External Systems page shows success rate, latency, last error, and recent call history.
10. Published artifacts contain credential references only, never secret values.
```
